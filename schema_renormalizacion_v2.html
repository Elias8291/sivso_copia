Mapeo Legacy → Normalizado

Lo que viene en concentrado (legacy)

Cada fila de concentrado representa 1 empleado + 1 artículo. El empleado se repite por cada artículo que se le asigna. Contiene:

  • Datos del empleado: NUE, nombre, apellidos, UR, delegación
  • Producto actual (post-muestreo): clave_partida, no_partida, descripción del producto, precio, cantidad, talla
  • Producto original (pre-muestreo): clave2025, otra descripción, otro precio_unitario, importe, IVA, total

El problema central: cada fila tiene DOS productos distintos — el que se planeó dar y el que realmente se dio.

Lo que viene en propuesta (legacy)

Catálogo maestro de productos. Cada fila tiene:

  • partida → es nuestro no_partida en la tabla partidas
  • partida_especifica → 244 o 246 (tipos en el rubro de egresos de Oaxaca) → tabla tipos_partida_especifica
  • Descripción, marca, unidad de medida, precio, código del producto

Esto mapea a: partidas + productos + producto_precios

Relaciones clave identificadas

  • Dependencia (UR) ↔ Delegación: N:M (una UR puede tener muchas delegaciones, una delegación puede pertenecer a muchas URs). Llave natural de dependencia = ur
  • Delegado ↔ Delegación: N:M (un delegado puede gestionar varias delegaciones, una delegación puede tener varios delegados)
  • Empleado → Dependencia (via UR) + Delegación: El empleado tiene NUE y UR. Cuando NUE es nulo, se compara UR + delegación para ubicarlo
  • Producto → Partida + Tipo Partida Específica (244/246)
  • Cupo Presupuesto: Gobierno asigna a cada dependencia un límite por partida + tipo. Se compara SUM(gasto real) vs límite
El problema de los dos productos en concentrado
── En concentrado, una fila típica ── Empleado: NUE=12345, nombre="Juan Pérez", UR=410, delegación="DEL-01" Producto ACTUAL (post-muestreo): Producto ORIGINAL (pre-muestreo): clave_partida = "UNIF-CAMP-M" clave2025 = "UNIF-CAMP-ORIG-M" descripcion = "Camisa campo azul" descripcion = "Camisa campo verde" precio_unit = 350.00 precio_unit = 420.00 no_partida = 2711 importe = 420.00 iva = 67.20 ────────────────────────────────────────────────────────────────── Son DOS productos distintos. El original se "compactó" al actual. Ambos deben mantenerse porque pueden volver a sustituirse.
Opciones de diseño para el producto dual

RECOMENDADA Opción A: Dos FKs en solicitudes_vestuario

Agregar producto_original_id (nullable FK → productos) a solicitudes_vestuario. El campo producto_id existente = producto final. Si hubo sustitución, producto_original_id ≠ producto_id.

+ Simple, 1 columna nueva | + Ambos productos viven en tabla productos con sus precios | + Fácil de consultar | + Soporta futuras sustituciones | + Sin cambio de flujo existente

− No guarda motivo ni fecha de sustitución (pero se puede agregar después)

Opción B: Tabla de sustituciones separada

Crear sustituciones_producto con: producto_original_id, producto_nuevo_id, anio, motivo, fecha. Y solicitudes_vestuario solo tiene producto_id (el final).

+ Historial completo de sustituciones | + Motivo y fecha de cada cambio

− Queries más complejas | − Sobre-ingeniería para el caso actual

Opción C: Tabla producto_versiones

Tratar cada representación como "versión" del mismo concepto de producto: tipo = original|muestreo|sustitucion.

+ Máxima flexibilidad

− Muy complejo | − Cambia la naturaleza de la tabla productos | − No vale la pena para 2 estados

Recomendación: Ir con Opción A ahora (simple, efectiva), y si en el futuro necesitas auditar motivos de sustitución, puedes agregar la tabla de sustituciones sin romper nada. Los dos enfoques son compatibles.

Cómo funcionan los dos productos

Regla de negocio

En concentrado, cada fila tiene dos conjuntos de datos de producto:

  • Producto actual (clave_partida): Lo que realmente se le da al empleado después del muestreo/compactación
  • Producto original (clave2025): Lo que originalmente se planeaba dar antes del muestreo

Ambos productos pueden ser sustituidos por código en el futuro, por lo que necesitamos mantener ambas formas y poder rastrear cambios.

ANTES Solo un producto por solicitud

-- Se pierde la info del producto original
solicitudes_vestuario
├── producto_id → productos  (¿cuál de los dos?)
└── No hay forma de saber qué se planeó dar

DESPUÉS Dos productos vinculados

-- Ambos productos coexisten
solicitudes_vestuario
├── producto_id          → productos  (FINAL)
├── producto_original_id → productos  (PLANEADO, nullable)
└── es_sustitucion       BOOLEAN

-- Consultas posibles:
WHERE es_sustitucion = true  → solo sustituciones
WHERE producto_id != producto_original_id  → idem
Tabla productos: alberga AMBOS tipos
── Importación desde concentrado ── 1. Leer clave_partida de concentrado → buscar o crear en productos → origen = 'propuesta' (viene del catálogo propuesta) → Este es el producto_id en solicitudes_vestuario 2. Leer clave2025 de concentrado → buscar o crear en productos → origen = 'concentrado' (producto pre-muestreo) → Este es el producto_original_id en solicitudes_vestuario 3. Si clave_partida == clave2025: → No hubo sustitución: producto_original_id = NULL, es_sustitucion = false 4. Si clave_partida != clave2025: → Hubo sustitución: ambos apuntan a productos distintos, es_sustitucion = true ── Sustitución futura por código ── 5. Crear nuevo producto en productos con origen = 'sustitucion' 6. En solicitudes_vestuario: → Guardar producto_id actual como producto_original_id → Actualizar producto_id al nuevo producto → es_sustitucion = true
Precios de cada producto
Cada producto tiene su propio precio: La tabla producto_precios almacena (producto_id, anio, precio_unitario). Tanto el producto original como el sustituto tienen entradas separadas. Así puedes comparar: "costaba $420 el original, se sustituyó por uno de $350".
Validación de presupuesto (cupos)
── Cómo funciona el control de gasto ── cupos_presupuesto: UR 410 + partida 2711 + tipo 244 + año 2026 → límite = $500,000 Para validar: SELECT SUM(sv.importe_total) FROM solicitudes_vestuario sv WHERE sv.anio = 2026 AND sv.dependencia_id = (dependencia con ur=410) AND sv.partida_id = (partida con no_partida=2711) AND sv.tipo_partida_especifica_id = (tipo con codigo=244) → Si total > límite → Excede presupuesto → Con FKs denormalizadas: INDEX SCAN puro, sin JOINs

Asignación de delegación cuando NUE es nulo
── Lógica de matching ── Caso 1: NUE no es nulo Empleado.NUE = concentrado.NUE → Buscar empleado existente por NUE → Si no existe, crear con su UR y delegación Caso 2: NUE es nulo → Tomar UR del empleado en concentrado → Buscar dependencia por UR → Buscar delegación que tenga esa misma UR (delegaciones.ur = concentrado.ur) → Comparar nombre completo para evitar duplicados → Asignar dependencia_id y delegacion_id ¿Por qué UR en delegaciones? Porque la delegación en concentrado tiene la UR del empleado. Al guardar delegaciones.ur, el match es directo: WHERE delegaciones.ur = empleado.ur AND delegaciones.clave = concentrado.delegacion
Cambios requeridos en la migración copiasivso
Estrategia: Como el esquema copiasivso se está construyendo desde cero (no hay datos legacy en producción), lo más limpio es reescribir la migración principal en lugar de agregar parches. Esto nos da un esquema limpio y consistente.

Resumen de cambios vs esquema actual

  • dependencias.ur: Cambiar de nullable()->index() a unique() — es la Business Key
  • delegaciones: Agregar columna ur (nullable, indexada) para facilitar matching
  • productos: Agregar origen ENUM('propuesta','concentrado','sustitucion')
  • solicitudes_vestuario: Agregar producto_original_id FK + es_sustitucion boolean + FKs denormalizadas

class SolicitudVestuario extends Model
{
    protected $connection = 'copiasivso';
    protected $table = 'solicitudes_vestuario';

    protected $fillable = [
        'empleado_id', 'producto_id',
        'producto_original_id',       // ← NUEVO
        'partida_especifica_id',
        'dependencia_id', 'partida_id', 'tipo_partida_especifica_id',  // ← denorm
        'anio', 'talla', 'cantidad',
        'precio_unitario', 'importe', 'iva', 'importe_total',
        'es_sustitucion',              // ← NUEVO
        'no_partida_snapshot', 'clave_partida_snapshot',
        'clave_para_ejercicio_snapshot',
        'legacy_concentrado_id',
    ];

    protected $casts = [
        'es_sustitucion' => 'boolean',
    ];

    // Producto FINAL (post-muestreo)
    public function producto(): BelongsTo
    { return $this->belongsTo(Producto::class); }

    // Producto ORIGINAL (pre-muestreo, nullable)
    public function productoOriginal(): BelongsTo
    { return $this->belongsTo(Producto::class, 'producto_original_id'); }

    // Relaciones denormalizadas (acceso directo)
    public function dependencia(): BelongsTo
    { return $this->belongsTo(Dependencia::class); }

    public function partida(): BelongsTo
    { return $this->belongsTo(Partida::class); }

    public function tipoPartidaEspecifica(): BelongsTo
    { return $this->belongsTo(TipoPartidaEspecifica::class); }

    // Relaciones originales
    public function empleado(): BelongsTo
    { return $this->belongsTo(Empleado::class); }

    public function partidaEspecifica(): BelongsTo
    { return $this->belongsTo(PartidaEspecifica::class); }

    // Helper: rellena FKs denormalizadas
    public function syncDenormFks(): void
    {
        $this->dependencia_id            = $this->empleado?->dependencia_id;
        $this->partida_id                = $this->producto?->partida_id;
        $this->tipo_partida_especifica_id = $this->producto?->tipo_partida_especifica_id;
    }
}
